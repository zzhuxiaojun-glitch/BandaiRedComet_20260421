# 11. 抢购当天操作手册

**用途**：从朋友说"我要抢这个"到实际下单成功的**完整操作流程**。每一步都给命令 + 期望输出 + 出问题怎么办。

---

## 11.1 预备阶段（朋友确定要抢后、开抢前 T-24h）

### 11.1.1 拿到抢购信息

问朋友 4 件事：

1. **商品链接 / 名字**：用来在小程序里找到它，拿 `spu_id`
2. **开抢时间**：精确到秒（例 `2026-05-10 20:00:00`）
3. **选哪个规格**：多规格商品（如有）需要朋友指定
4. **限购 N 件**：商品详情页有写，超限 createOrder 会直接拒

### 11.1.2 拿商品 ID

**方式 A（推荐，朋友会操作就用）**：让朋友在小程序里找到商品 → 截图商品详情页 URL
- 万代 PB 商品 URL 形如 `pages/commodity/commodity?id=6521`，那个 `id=6521` 就是 `spu_id`

**方式 B（朋友不会）**：按 [`3_抓包指南.md`](./3_抓包指南.md) 抓一次朋友小程序的 HAR，在 HAR 里搜 `spu/v2/detail/new?encryptionUrlParams=` → 用我们的 `tools/verify/dump_full.py` 解出 id

**拿 sku_id**：脚本跑 `check` 后，日志会打印商品信息里的 `skuList`。或者直接调 `/api/commodity/v1/app/spu/sku/detail?id=<spuId>`（tools/verify/dump_full.py 同一 HAR 里找一下）。

### 11.1.3 让朋友给一个"新鲜 CK"

CK = 他微信登录万代小程序后的 api-access-token。按 [`10_使用指南.md` §10.2](./10_使用指南.md) 拿。

**抢购前 1-2 天拿一次**足够用，JWT 一般有效期几天，看了 HAR 里的 exp 字段可以精确判断。

### 11.1.4 填 config.yaml

基于 `config.example.yaml` 复制一份 `config.yaml`，填入：
- `ck`（朋友给的）
- `snipe_time`（开抢时刻）
- `target.spu_id` / `target.sku_id` / `target.num`（几件）
- `target.address_id` —— 跑一次 check 让脚本打印朋友账户的地址 id 列表，选一个

## 11.2 T-1 天：预检 + 通知测试

### 11.2.1 跑一次 check

```bash
PYTHONPATH=src .venv/bin/python -m bandai_sniper check
```

看到 4 个 ✅：
```
✅ CK 有效, memberId={XXXXXX}
✅ 商品「...」 price=... stock=... saleStatus=...
✅ 地址 id=... ...
SUCCESS 预检通过
```

**常见问题**：
| 现象 | 原因 | 处理 |
|---|---|---|
| `HTTPStatusError 432` | UA/Referer 没带 | 确认 `client.py` 的默认头没被误改 |
| `code=302/703/307/1004` | CK 过期 | 让朋友**立即**重抓一次 CK 粘进来 |
| `地址 id=X 在地址列表里找不到` | config 里 address_id 写错 | 用预检日志里的真实 id 替换 |
| `商品查询 saleStatus != 0` | 未开售 | **正常** —— 抢购前肯定没开卖 |

### 11.2.2 测通知通道（防止抢到了但没收到推送）

朋友爱用哪个推送方式？选一个配进 `config.yaml` 的 `notify` 段：
- **server 酱**（最简单）：https://sct.ftqq.com 拿 SCTKEY
- **Bark**（苹果用户）：App Store 装 Bark，拿个人推送 URL
- **飞书机器人**：拉朋友进一个群 → 添加自定义机器人 → 拿 Webhook

然后试发一条 dummy：

> 当前脚本没有 `notify-test` 子命令；**临时方案**：在 Python 里手动跑一次
> ```python
> import asyncio
> from bandai_sniper.config import load_config
> from bandai_sniper.notify import push
> cfg = load_config("config.yaml")
> asyncio.run(push(cfg.notify, "测试", "这是抢购前的通知测试"))
> ```
> 朋友收到推送 = 通道 OK。**TODO**：后续加个 `python -m bandai_sniper notify-test` 子命令更方便。

### 11.2.3 NTP 对时

```bash
PYTHONPATH=src .venv/bin/python -m bandai_sniper ntp
```

看本机偏差：
- `< ±100ms`：本机时钟准
- `< ±500ms`：脚本会内部校准，够用
- `> ±1s`：考虑让朋友那台机也跑一下或者换机器

## 11.3 T-30 分钟：最后确认

### 11.3.1 再跑一次 check（防 CK 突然过期）

```bash
PYTHONPATH=src .venv/bin/python -m bandai_sniper check
```

CK 要是在这一步突然挂了，立刻让朋友重抓并更新 config。

### 11.3.2 确认 `snipe_time` 没写错

`date` 命令看当前时间，和 `snipe_time` 对比。**时区尤其要确认**：config 里是 `Asia/Shanghai`，朋友如果在别的时区就会偏。

### 11.3.3 准备看日志的窗口

开两个终端：
- 主终端：抢购时跑 run
- 副终端：`tail -f logs/run_*.log` 看实时日志流（脚本会自动落盘）

## 11.5 抢到后：让朋友付款

**现阶段**：脚本只返回 `prepay_id`，没法直接唤起微信支付（小程序专属 API，Python 调不了）。朋友必须**在微信里付款**：

1. 打开万代小程序
2. **我的 → 我的订单 → 找到刚才那笔待支付**（订单号在日志里）
3. 点"去支付" → 走微信支付流程 → 扫脸 / 输密码 / 指纹

**15 分钟内必须付款**，否则订单自动取消、库存释放给下一个人。

> **TODO** 优化：抢到后 stdout 直接打 QR 码（`qrcode` 库 10 行代码），朋友手机扫码更快。
> 但微信小程序支付走 JSAPI（必须在 WebView 内），QR 形式支付实际走的是"微信二维码支付"，和 JSAPI 的 prepay_id 不通用，**做不了**。还是让朋友进小程序付。

## 11.6 抢失败：怎么查原因

### 11.6.1 看日志最后几行

```bash
ls -t logs/ | head -3   # 最新的三个 log 文件
tail -50 logs/run_<最新>.log
```

### 11.6.2 常见失败码

| 业务 code | 含义 | 处理 |
|---|---|---|
| `1` | 通用业务错（msg 会具体说） | 看 msg，可能是库存不足 / 限购 / 未开售 |
| `2001` | 被风控拒 | 等几分钟，换设备 / IP 再试 |
| `101` | 参数错 | 脚本 bug，检查 confirmOrder 响应结构 |
| `302/703/307/1004` | 登录态失效 | CK 过期，重抓 |

如果 log 里 N 个 worker 全部失败，脚本会打 `全部 worker 重试失败 · 最后: ...`，把那个错误 code 加到 `strategy.retryable_codes`，下次才会重试那个错。

### 11.6.3 抢不到就是抢不到

万代 PB 瞬爆款是真瞬爆，脚本能把"人类点击 → API 调用"的延迟从 ~2s 压到 ~150ms，但服务端本身有上限（大概 1s 内接完限额就关闸）。**能做的 = 减少自己这边的无谓延迟 + 尽量靠近服务器发请求（用 WSL 比用朋友家里 WiFi 快）**。

## 11.7 抢购前的"四件必做事"清单（对应截图要求）

这些当下就能做（不用等抢购）：

### [1] 真打一次有时间压力的抢购测试
**现在做**：用 `tools/smoke/test_real_order.py` 跑一次真 createOrder（已跑过，订单 1416249 已取消释放），**验证过的**。

如果还想测 countdown + fire 流程，**没法模拟**"到点触发"除非真有抢购。退而求其次：改 `snipe_time` 为 30 秒后的时刻，跑 `run`，让它从 T-30s 一路走到 T=0，就能看到 countdown + fire 能不能连起来。**但这会真创建一笔订单**（虽然会被 15 min 自动取消）。

### [2] 填 retryable_codes
**等实战抓到码再做**。真抢购时肯定会碰到几个"未开抢 / 限流 / 库存争抢"类型的 code，抓到后加到 config。

**现在做**：按 §11.2.2 的 Python 小脚本跑一下。
**后续优化**：加 `python -m bandai_sniper notify-test` 子命令。

### [4] 支付 QR 码
**做不了**：微信小程序支付只支持小程序内唤起 JSAPI，脚本拿到的 `prepay_id` 没法转成独立 QR 码。朋友只能在微信小程序里手动付。

这四条里 [1] 和 [3] 我们能立刻做，[2] 要等实战，[4] 是技术限制。
