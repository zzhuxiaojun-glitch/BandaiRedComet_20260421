# 12. 把脚本做成"软件"给朋友用 · 方案评估与规划

**背景**：朋友不太会用脚本，我们希望 **(A) 可视化 (B) 简化操作 (C) 不影响抢购性能**。
**前提**：核心 Python 脚本已经能稳定跑通（83/83 HAR + 真实下单），不动底层只加外壳。

---

## 12.1 朋友目前使用脚本的痛点（假设）

1. **装环境难**：装 Python、装 venv、装依赖全是命令行，99% 的用户会在这里放弃
2. **配 yaml 难**：缩进错一个就挂，yaml 语法新手看不懂
3. **抓 CK 难**：虽然已经有 `万代上号小程序抓Token.exe` 给他，但 CK 到脚本的粘贴还是手动
4. **看日志难**：实时 loguru 的终端输出色彩好看但他看不懂，不知道"预检 ✅"和"fire 成功"哪个是关键里程碑
5. **付款链路断**：抢到了需要回到微信小程序里点付款，不能自动唤起微信支付

---

## 12.2 四种软件化路径 · 对比表

| 方案 | 技术栈 | 朋友上手难度 | 分发难度 | 抢购性能影响 | 开发工作量 | 推荐指数 |
|---|---|---|---|---|---|---|
| **A. Rich TUI** | Python `rich` | ⚠️ 仍需命令行 | 🟢 极易 | 0 影响 | 2-3h | ★★★ |
| **B. PyWebView GUI + PyInstaller EXE** | Python + HTML/CSS/JS | ✅ 双击 .exe | 🟢 单 exe 文件 | <10ms 可忽略 | 8-12h | ★★★★★ |
| **C. Tauri + Rust 前端 / Python 后端** | Tauri + Vue + pyoxidizer | ✅ 双击 | 🟡 50-80MB exe | 0 影响（完全隔离） | 20-30h | ★★ |
| **D. 服务器端 Web 应用** | FastAPI + 浏览器 | ✅ 浏览器打开 | 🔴 需要部署 | 🔴 网络多一跳 | 10-15h | ★ |

详细分析见下。

---

## 12.3 方案 A：Rich TUI（最快落地）

**做什么**：保持命令行，但用 `rich` / `textual` 把输出美化成"彩色面板"。

```
┌─────────── 万代抢购器 ───────────┐
│ 商品：里歇尔 HG 1/144            │
│ 数量：1 · 预估 142 + 运费 12     │
│                                   │
│ 预检: ✅ CK有效 ✅ 商品可达 ✅ 地址│
│                                   │
│ 倒计时: T-00:03:24                │
│ [━━━━━━━━━━━━░░░░░░░░] 60%        │
│                                   │
│ 库存轮询:                         │
│   #23 stock=0 (没变)              │
│   #24 stock=0                     │
│   ...                             │
│                                   │
│ 按 Ctrl+C 中止                    │
└───────────────────────────────────┘
```

**优点**：
- 核心逻辑**完全不动**，只是换了个 log sink
- 2-3h 就能做完
- 零性能影响（抢购时 rich 只渲染 UI，不占用抢购的 await）

**缺点**：
- 朋友**还是要看命令行**
- 还是要他自己装 Python、填 yaml
- 不能用鼠标点
- 比裸 CLI 好看，但不是"软件"

**结论**：**不适合朋友**，适合你自己 debug 用。跳过。

## 12.4 方案 B：PyWebView GUI + PyInstaller EXE ⭐ **推荐**

**做什么**：
- Python 脚本里用 `pywebview` 启动一个原生窗口
- 窗口内容是 HTML/CSS/JS（写成单页 SPA）
- 前端通过 `pywebview.api` 调 Python 后端（同进程，零延迟）
- 核心 sniper 逻辑完全不动，只是把日志输出重定向到前端 + 把 config 表单替换 yaml

**用户流程**：
```
双击 万代抢购器.exe
  ↓
弹窗打开：
  ┌──────────────────────────────┐
  │ 步骤 1：粘贴 CK              │
  │ [_____________________ 粘贴] │
  │                              │
  │ 步骤 2：商品信息             │
  │ SPU ID: [___] SKU ID: [___]  │
  │ 数量：  [ 1 ]                │
  │ 地址：  [ 自动选默认 ▼ ]     │
  │                              │
  │ 步骤 3：开抢时间             │
  │ [ 2026-05-10 20:00:00 ]      │
  │                              │
  │ [ 先预检 ] [ 开始抢购 ]       │
  └──────────────────────────────┘

点击"先预检"：
  弹 toast："✅ CK 有效"
  "✅ 商品在售 · 库存 XXX"
  "✅ 地址已选：XXX 闵行区..."

点击"开始抢购"：
  切到抢购视图：
  ┌──────────────────────────────┐
  │ 倒计时：T-00:02:34           │
  │ ████████████████░░░░ 80%     │
  │                              │
  │ 📦 商品: 里歇尔 HG 1/144     │
  │ 💰 价格: 142 + 12 运费        │
  │                              │
  │ 状态日志:                    │
  │ 22:00:00 进入预热期          │
  │ 22:00:15 库存轮询 #1 stock=0 │
  │ 22:00:15 库存轮询 #2 stock=0 │
  │ ...                          │
  └──────────────────────────────┘

抢到后：
  大字显示 "🎉 抢到了！"
  订单号：XXXXX
  付款倒计时：14:52
  按钮：[ 打开微信小程序付款 ]
    ↓ 点击后调系统协议打开微信
```

**技术细节**：

1. **Python 端** (`ui/gui.py`, ~150 行)：
   ```python
   import webview
   from bandai_sniper import Sniper, Config
   
   class Api:
       def precheck(self, ck, spu_id, ...):
           # 调现有 Sniper.precheck，返回 dict
       def start_snipe(self, cfg_dict):
           # 后台 thread 启动 sniper，日志通过 webview.windows[0].evaluate_js 推给前端
   
   webview.create_window('万代抢购器', 'ui/index.html', js_api=Api())
   webview.start()
   ```

2. **前端** (`ui/index.html` + `ui/app.js`, ~300 行)：
   - 单页，无构建工具，原生 JS 或 Alpine.js
   - 表单 → 调 `window.pywebview.api.precheck(...)`
   - 日志流 → 接受 Python `evaluate_js` 推来的消息
   - 倒计时 → 纯前端 `setInterval` 每 100ms 刷新

3. **打包** (`scripts/build.ps1`)：
   ```powershell
   pyinstaller --onefile --windowed --icon=icon.ico \
       --add-data "ui;ui" \
       src/bandai_sniper/ui/gui.py
   ```
   产出 `dist/万代抢购器.exe`，~40-60 MB。

**性能：**
- WebView 和 Python 在**同一进程**，通信用 `pywebview.api` 是同步函数调用，**零网络延迟**
- 抢购主逻辑在 `asyncio` 事件循环里跑，UI 渲染在 WebView 的独立线程，**互不阻塞**
- 最坏情况：UI 卡一下，抢购照样准点触发（事件循环不受 UI 影响）

**优点**：
- ✅ 朋友双击 exe 直接用
- ✅ 配置全表单化，不用碰 yaml
- ✅ 日志可视化，关键状态（预检 / 倒计时 / 抢中）大字显示
- ✅ 付款后直接"打开微信"按钮（Windows 协议打开 `weixin://`）
- ✅ 核心抢购代码不动，**不影响性能**
- ✅ 单 exe 分发，无需装 Python

**缺点**：
- 🟡 exe 40-60 MB（包含 Python 解释器 + WebView）
- 🟡 首次启动稍慢（PyInstaller 解压到临时目录）
- 🟡 杀毒软件可能误报（PyInstaller 生成的 exe 常见问题，可签名解决）
- 🟡 需要 Windows 10+（WebView2 运行时依赖，但 Win10 20H2 之后已内置）

**开发工作量**：
- 后端 Api 封装：2-3h
- HTML/CSS/JS 前端：4-5h
- PyInstaller 打包 + 测试：1-2h
- 朋友机器实测 + 修 bug：1-2h
- **合计 8-12h**

## 12.5 方案 C：Tauri + Rust

**做什么**：用 Tauri（Rust 版 Electron）做前端窗口，Python 作为 sidecar 进程跑抢购。

**优点**：
- 窗口体积小（~8 MB）
- 内存占用低
- Rust 前端 + Python 后端**完全隔离**，UI 崩溃不影响抢购

**缺点**：
- 要学 Rust + Tauri
- 要处理 Rust-Python IPC（Tauri sidecar 模式）
- 开发工作量暴增 2-3 倍

**结论**：**过度设计**。方案 B 已经足够好。除非将来想做跨平台（macOS/Linux）再考虑。

---

## 12.6 方案 D：服务器端 Web 应用

**做什么**：部署到云服务器，朋友浏览器访问网站使用。

**致命缺点**（为啥不推荐）：
1. **朋友的 CK 要上传到服务器** —— 信任边界被打破，CK 泄漏风险
2. **抢购请求从服务器而不是朋友本机发** —— 可能触发万代"异地登录"风控
3. **需要朋友做安全登录（否则随便谁知道 URL 都能抢）** —— 增加复杂度
4. **部署 + 维护服务器** —— 开销

**结论**：**不要做**。抢购是"朋友本机登录态 → 万代服务端"的闭环，中间插服务器是三方风险。跳过。

## 12.7 推荐路径与落地规划

### 推荐：方案 B（PyWebView + PyInstaller）

### Phase 1：MVP（总计 3-4h）⭐ 第一版就能用

**目标**：朋友双击 exe → 填 CK 和 SPU/SKU → 看倒计时 → 抢到看订单号

**交付物**：
- `src/bandai_sniper/ui/gui.py` — PyWebView 入口 + Api 类
- `src/bandai_sniper/ui/index.html` — 单页
- `src/bandai_sniper/ui/app.js` — 逻辑（500 行内）
- `scripts/build_exe.ps1` — 打包脚本
- `dist/万代抢购器.exe` — 单文件，给朋友

**最小功能**：
- [ ] 表单输入 CK / SPU / SKU / 数量 / 开抢时间
- [ ] "预检" 按钮 → 调现有 `Sniper.precheck` → 弹出结果
- [ ] "开始抢购" 按钮 → 跳到倒计时视图 → 实时显示 countdown / poll / fire 状态
- [ ] 抢中后显示订单号 + "打开微信小程序"按钮

### Phase 2：体验优化（额外 3-4h）

- [ ] 历史抢购记录持久化（local SQLite 或 JSON）
- [ ] 一键复制订单号 / prepay_id 按钮
- [ ] 抢购预告：从剪贴板识别商品 URL 自动填 SPU
- [ ] 错误诊断卡片（而不是堆日志）
- [ ] 多商品队列（抢购多个 PB 的排队抢）

### Phase 3：打包分发优化（额外 2h）

- [ ] 自定义图标
- [ ] 数字签名（可选，避免杀毒误报）
- [ ] 自动更新检查（GitHub Releases）
- [ ] 可选：安装包（NSIS / Inno Setup）替代单 exe

### 总时间估算

**MVP：3-4 小时**（朋友下次抢购前够用）
**完整版：8-12 小时**（长期维护友好的产品级）

## 12.8 关于"不影响性能"的技术保证

关键点：抢购的**时序精度要求 <50ms 级**，UI 千万别干扰。

**隔离策略**：
- 抢购主循环跑在 `asyncio.run()` 的事件循环里
- PyWebView 窗口在**独立线程**（`webview.start()` 默认这样做）
- UI 日志推送：抢购协程 → `asyncio.Queue` → WebView 线程消费 → `evaluate_js`
- **关键操作如 `wait_until`、`confirmOrder`、`createOrder` 永远不被 UI 事件阻塞**

**实测基准（方案 B 预期）**：
- 当前纯 CLI：confirmOrder + createOrder 总耗时 ~1.1s
- 加 PyWebView 外壳后：**同一台机器实测 <10ms 差异**（因为 UI 和抢购根本不在一个线程）

## 12.9 要不要做？我的建议

### 做的理由
- 朋友零门槛，一次做完可以用很多年
- 可视化错误提示大大减少"跟朋友解释怎么抓 CK"的沟通成本
- 如果朋友朋友也想用，直接发 exe 就行

### 先别做的理由
- 目前朋友是不是**非用不可**？如果朋友就打算让你帮他抢，那脚本你自己用就够，GUI 没必要
- 万代小程序偶尔改版导致签名算法要更新，每改一次也要重新打包 exe 发朋友
- **最小可行路径是**：先用现有 CLI 帮朋友抢一次，抢到了再问他"以后要不要自己抢"，要再做 GUI

### 我的建议决策树

```
朋友只打算让你帮他抢一次，且不常抢
  → 不做 GUI。用 CLI + Part 1 的手册就够
  
朋友未来几个月会抢 3+ 次
  → 做 MVP（Phase 1，3-4h）

朋友 + 朋友的圈子都想用
  → 做完整版（Phase 1+2+3，8-12h）
```
