# 16. 真抢购演练 SOP（GUI · 小价格商品）

**目的**：用一个**已开售的低价商品**走完完整 GUI 抢购流程到拿 `prepay_id` 为止，验证 GUI 整套链路真能用，**不付款让订单 15 min 自动取消**。

**为什么要做**：
- 之前的真订单 `1416249` 是 CLI 跑的；GUI 的 precheck / countdown / fire / 抢中视图 / 错误卡片**还没用真订单走过**
- 顺便拿到实战业务码填进 `strategy.retryable_codes`
- 顺便测通知通道（钉钉/Server酱/飞书）

**预计耗时**：30-60 min（含等 snipe_time 触发）

**前置假设**：完成过 `14_GUI测试清单.md §14.0 环境准备`（pyqt6 + WebEngine + CJK 字体已装好）

## 16.3 演练步骤（按顺序执行）

### 步骤 1：填账号

| 操作 | 预期 |
|---|---|
| 把 CK 粘到"① 账号" 输入框 | （175 字符 JWT） |
| 勾选"记住 CK"（可选） | 下次省得粘 |

### 步骤 2：选商品（用 HAR picker）

| 操作 | 预期 |
|---|---|
| 在"② 商品" 卡片里点"📁 从 HAR 选商品" | 弹出原生文件选择器 |
| 选 `bandai_capture_20260426_01.har` | Modal 弹出 9 个商品列表 |
| 点 SPU 1268 那一条（¥39 电王腰带） | Modal 关闭、SPU 输入框自动填 `1268` + 闪绿 ✓ + feedback "已选商品 SPU=1268..." + 自动拉 SKU |
| 看 SKU 下拉 | 应该有 1 条（这商品规格简单） |

**预检验证**：
| 操作 | 预期 |
|---|---|
| 点"先预检" | 蓝色 "验证中..." → 绿色 "✅ 预检通过 · memberId=796529 · N 个地址" |
| 看商品预览框 | 黄底显示中文名 + 日文（如有）+ 价格 ¥39 + 库存 + "可售" |
| 看地址下拉 | 自动填好你账户的地址 |

**如果预检挂**：
- 红色卡片显示 🔑 "CK 已失效" → 重新抓 CK
- 红色卡片显示 📡 "网络超时" → 检查网络
- 其他 → 看错误卡片提示

### 步骤 3：选数量、地址

| 操作 | 预期 |
|---|---|
| 数量填 `1` | 默认就是 1 |
| 地址下拉选你正确的家庭地址 | （就 1 个一般） |

### 步骤 4：设 snipe_time

**关键**：不用真等 PB 抢购窗口，**设个 60 秒后的时间**：
- 在"④ 开抢时间"输入：当前时间 + **1 分钟**
- 例：现在 22:30:00 → 填 22:31:00

**为啥要等 60 秒**：让 GUI 走完完整 countdown 流程（预热 → poll 库存 → 准点 fire），不是即时触发。

### 步骤 5：发射

| 操作 | 预期 |
|---|---|
| 点 🔥 "开始抢购" 红色大按钮 | 0.5 秒内切到"抢购中"视图 |
| 看倒计时 | 大字 `T-00:00:55` 之类，每 100ms 刷新 |
| 看进度条 | 从 0% 缓慢涨到 100%（夏亚红 → 夏亚粉渐变） |
| 看实时日志 | 黑底彩字，能看到：sync_timestamp / whoami / spu_detail / poll baseline / poll #1..N / countdown / fire |
| 中途想停 | 点"中止"应该能切到失败视图（"已手动中止"图标） |

### 步骤 6：T=0 触发

观察发生顺序：
1. 倒计时归零，阶段从"等待开抢"切到"下单中"
2. 日志里出现 `[#0] confirmOrder` → `[#0] draft orderAmount=39.0...` → `[#0] createOrder`
3. **0.5-2 秒后**应该看到 SUCCESS 日志：`🎉 抢单成功 · order_id=XXXXX prepay_id=wx...`
4. GUI 切到金色 "抢 到 了" 成功视图

**如果失败**：
- 走到失败视图 → 看错误卡片图标和提示
- 截图 + 把错误卡片的"原始错误"展开内容贴给我
- 把当时**业务 code（如果有）记下来** → 后面填进 retryable_codes

### 步骤 7：成功视图操作

| 操作 | 预期 |
|---|---|
| 看大字 "抢 到 了" | 金黄色 + 字距大 |
| 看订单 id | 像 `1416XXX` 的整数 |
| 看 prepay_id | `wx22XXXXXXXX` |
| 点"复制"按钮（订单 id 那行） | feedback "已复制" |
| 把订单 id 粘进任意位置看 | 真的复制了 |
| 点"打开微信"按钮 | Windows 下应该能调起微信（如不行也不阻塞 —— 已知功能） |
| 点"再抢一个" | 回表单视图 |

### 步骤 8：去微信验证订单存在

打开万代小程序 → 我的订单 → 看到一笔 **`BD26XXXXXXXX` "待支付"** ¥39 的订单，是步骤 6 创建的。

**确认**：订单存在 + 商品名 + 价格都对 = 演练完美成功。

### 步骤 9：等订单自动取消

**不要付款**。订单 15 min 内自动取消，库存释放给下一个人。

或者主动操作：万代小程序 → 我的订单 → 那笔订单 → "取消订单"（更利他）。

### 步骤 10：通知验证（如果步骤 16.2.3 配了通道）

抢中那一刻你的手机应该收到 1 条推送 "万代抢单成功 · 订单 id=XXX..."。**没收到** = notify 通道有问题，演练后修。

## 16.5 异常情况预案

### 5.A 演练时电王腰带卖光了

库存 27 -> 0 是可能的。换个 SPU：
- `1434` HG 帝王异端Ω ¥130（库存 153）
- `6240` EG ν 高达 ¥145（库存 647）
- `6573` 海贼王艾尼路 ¥799（库存 995）

但价格越高，万一手抖付了越心疼。**优先 1268，挂了换 1434**。

### 5.B 抢的时候触发风控

如果遇到 `code=2001` 之类的风控提示：
1. 等 5-10 分钟
2. 重启 GUI
3. 重抓 CK 再试

**这是宝贵的实战数据**：把那个 code 加到下次的 `strategy.retryable_codes` 里。

### 5.C CK 演练当中过期

中间出现 🔑 "CK 已失效"：
1. 重新抓 HAR 拿新 CK
2. 替换 GUI 输入框
3. 重新跑步骤 5

### 5.D 倒计时过了但 fire 没触发

如果 T=0 之后 5 秒还没看到 fire 日志：
- 看 Web Inspector → Console 有没有 JS 错误
- 看终端有没有 Python 异常
- 截屏给我，可能是异步逻辑 bug

## 16.6 演练成功标准

满足以下三件事算演练通过：

1. **GUI 切到金黄色"抢 到 了"视图**（说明 fire / createOrder 在 GUI 链路里成功）
2. **微信小程序里能看到那笔订单**（说明真到了万代服务器）
3. **复制 prepay_id 成功**（说明数据在 UI 里正确填充）

可以不满足的（不阻塞验证通过）：
- 通知没收到（看 5.E 调）
- "打开微信"按钮在 WSLg 下不工作（Windows 下才有 weixin:// 协议支持）
