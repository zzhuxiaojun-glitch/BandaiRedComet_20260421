## 一、v1.4 主任务：全自动抢购模式

### 朋友的原话

> 现在是自动能搜出来，但是得我手动点一下，再点开抢
> 然后我设的时间是15点的话，会加1fz再抢
> 就得要搜出来，点击，加直接抢，这一串得自动

### 用户痛点拆解

当前（v1.3）流程：
```
填关键词 → 勾自动轮询 → 命中通知 → ⛔️手动点搜索结果选商品 → ⛔️手动点开始抢购 → 抢
```

朋友期望（v1.4）：
```
填关键词 → 勾全自动模式 → 启动 → ✅自动命中 → ✅自动选 SKU → ✅自动到点开抢 → 抢
```

### 设计决策（已确认）

| # | 决策点 | 最终方案 | 备注 |
|---|---|---|---|
| 1 | 命中多个商品选哪个？ | **选第一个**（万代搜索按相关性排序，第一个最匹配） | — |
| 2 | 多 SKU 怎么选？ | **单 SKU → 自动选第一个最匹配；多 SKU → 不抢，弹警告** | 朋友确认"一般要抢的东西只有一个选项" |
| 3 | 命中时机？ | 未到开抢时间 → 等到准点抢；到了开抢时间但还没命中 → 命中即抢；超过开抢+N 分钟还没命中 → 停止+提示 | N 默认建议 5 分钟 |
| 4 | 安全开关 | 启动时弹确认对话框，措辞改为"占库存"而非"扣钱"："⚠️ 全自动模式将自动选商品 + 自动创建订单（占库存）。订单需在 N 分钟内手动微信支付，否则自动取消。确认？" | 不真扣钱，支付仍人工 |
| 5 | 中途能停？ | "🛑 中止"按钮全程可点 | — |

### 状态机设计

```
        ┌──────────────┐
        │   IDLE       │ 用户填表
        └──────┬───────┘
               │ 点「启动全自动」
               ▼
        ┌──────────────┐
        │ AUTO_POLLING │ ←─┐ 每 N 秒搜
        └──────┬───────┘   │
               │ 命中       │ 没命中
               ▼            │
        ┌──────────────┐   │
        │AUTO_SELECTING│   │ ──→ 看 SKU 数量
        └──────┬───────┘
               │ 单 SKU
               ▼
        ┌──────────────┐    多 SKU
        │   FIRING     │   ──→ ABORTED + 弹警告
        └──────┬───────┘
               │
               ▼
        ┌──────────────┐
        │SUCCESS/FAILED│
        └──────────────┘

特殊路径：
- 任意状态用户点「中止」 → STOPPED
- AUTO_POLLING 超过 (开抢时间 + N 分钟) → STOPPED + 提示
```

## 二、抢购速度优化清单（与 v1.4 解耦，可独立做）

朋友说"越快越好，这才是抢购的意义"。代码读透后定位到的优化空间——

### 🔴 高优先级（最大瓶颈）

#### 1. 填准 `retryable_codes`

**位置**：`config.py:25` `Strategy.retryable_codes`，默认 `[]`

**问题**：
- 默认空 → 任何业务错误码（包括"未开售"）一打就立刻 fail，不重试
- 万代后端常见在标称开抢时间晚 0~500ms 才真正开闸
- 脚本在准点 fire，第一次拿到"未开售" → worker 立即放弃 → fire() 整体 RuntimeError 退出
- 用户感受：脚本"准点抢但抢不到"，怀疑是不是延后抢的

**修复**：实战抓一次"未开售"业务码（比如 `2003` / `1001` 之类），加进去：
```python
retryable_codes = ["2003", "1001"]  # 实测后填
```

**怎么抓**：让朋友 v1.4 实战时打开 logs，开抢瞬间日志会有 `ApiError code=XXX msg=未开售` 之类，记下来。

---

#### 2. v1.4 全自动模式自带"日志一键复制路径"

**位置**：失败错误卡片底部（前端）

**改进**：抢购失败 → 错误卡片直接显示"📋 复制日志路径"按钮，点了把 `logs/gui_xxxxxx.log` 完整路径复制到剪贴板。朋友直接粘贴发给我就能定位。

**收益**：不用每次教朋友"打开文件夹找最新一个日志"。

---

### 🟡 中优先级

#### 3. `lead_ms` 默认值偏保守

**位置**：`config.py:19`，默认 `lead_ms=5`

**问题**：5ms 是 WSL 测的；Windows 物理机 + 国内线路（万代上海机房）TTFB 通常 30-80ms，5ms 太少。

**修复**：默认改 `15ms`（Win 物理机更合理），UI 加滑块让用户实战调（10/20/30/50ms 几档）。

---

#### 4. `max_early_fire_ms` 默认偏保守

**位置**：`config.py:35`，默认 `max_early_fire_ms=2000`

**问题**：万代瞬爆款经常提前 1-3 秒就把 stock 写进去了，2 秒提前量可能错过早开放窗口。

**修复**：默认改 `5000`（5 秒），但需要实战验证不会触发风控。

---

#### 5. `concurrency` 是否调高

**位置**：`config.py:21`，默认 `concurrency=3`

**问题**：3 路并发很保守。万代风控阈值未知，调到 5-8 可能更激进，但也可能触发限流。

**修复**：先实测当前 3 路是否触发风控，再决定要不要加。**不要盲目调高**。

---

### 🟢 低优先级（产品级体验）

#### 6. NTP 失败时降级提示更明确

**位置**：`timer.py:25` `"NTP 全部失败，使用本机时间"`

**问题**：NTP 全部失败时只是 warning，用户可能没注意。本机时钟如果慢几秒，准点 fire 实际就是延后几秒。

**修复**：NTP 全失败时在 GUI 顶部弹**红色警告条**："⚠️ NTP 校准失败，本机时钟可能不准，建议手动校时后再抢"。

---

#### 7. `pre_warmup_seconds` 60s 的合理性

**位置**：`config.py:18`，默认 `pre_warmup_seconds=60`

**讨论**：60 秒预热建 keep-alive 够用，但万代风控可能监测"长 keep-alive + 准点突发"模式。如果是的话，预热反而帮倒忙。需要实战观察是否被风控。

---

#### 8. 多线 NTP 服务器并发对时

**位置**：`timer.py:17` 当前是顺序遍历

**问题**：第一个 NTP 失败要等 timeout，浪费几秒。

**修复**：改用 `asyncio.gather`/`wait` 并发请求多个 NTP 服务器，取**最先返回**的结果。

---

#### 9. 抢购成功后自动唤起微信小程序付款

**讨论**：抢到后给微信链接 / 二维码 / Server 酱推送，让朋友尽快支付（15 分钟超时即丢单）。

**当前**：v1.0 已有 PushPlus 推送（`notify.py`），但没自动唤起小程序。

**改进**：能不能用万代小程序的 schema URL 直接打开订单付款页？需要抓包确认。

---

#### 10. confirmOrder + createOrder 串行 vs 并行

**位置**：`sniper.py:181-194` `_attempt()`

**讨论**：当前是 confirmOrder → createOrder 串行。这两步必须串行（createOrder 依赖 confirmOrder 返回的 orderAmount 等字段），但 confirmOrder 可以**预先在预热阶段打一次**（提前算价 + 缓存 draft），开抢时只打 createOrder，理论上能省 1 个 RTT。

**风险**：缓存的 draft 如果包含"未开售"状态，createOrder 可能失败。需要确认万代 confirmOrder 接口在未开售时的返回结构。

**收益**：~50-200ms。瞬爆款里这点很关键。

## 三、其他可改进点（这次会话挖出来的）

### 11. `runtime_hooks` 文件路径用 `os.path.join` 跨平台

**位置**：`万代抢购器.spec:58` `runtime_hooks=["scripts/pyi_rth_pythonnet.py"]`

**问题**：写死了正斜杠，PyInstaller 在 Win 上一般也认，但不是最稳。

**修复**：用 `os.path.join` 或 `pathlib.Path` 拼路径。

---

### 12. `_check_ascii_path` 兜底范围扩展

**位置**：`main.py:24` `_check_ascii_path()`

**当前**：只检查 exe 所在目录非 ASCII。

**扩展**：也检查 `%TEMP%`、`%APPDATA%`、用户名等 pythonnet 可能引用到的路径。但这些通常用户改不了，所以优先级低。

---

### 13. classifyError 兜底类目分级

**位置**：`app.js:652` `classifyError()`

**当前**：fallback 是 "异常 · 未分类错误"，没分级。

**改进**：给 fallback 也分级（看错误字符串里有没有 "Error" / "fail" 关键字），不要一律 "error" 严重级别。

---

### 14. 日志 redact 范围

**位置**：`client.py:111` 日志写整个请求/响应

**问题**：完整 access-token 虽然 redact 了（`eyJ0eX***sUpE`），但收货人姓名、电话、地址还是明文写进 logs。

**改进**：
- `receiverPhone` 写中间 4 位掩码
- `receiverAddress` 不变（debugging 需要）
- 加配置 `log.redact_pii=True/False` 让用户选

---

### 15. dist/ 应该 .gitignore

**问题**：dist/ 现在是空的（脚本会清），但万一打包失败留下半成品，会被 git tracking 污染。

**修复**：确认 .gitignore 里有 `dist/`、`build/`、`.venv-win/`。

## 四、决策日志（v1.4 设计中已经否决的方案）

| 方案 | 否决原因 |
|---|---|
| 命中即抢（不等准点） | 朋友确认"未到开抢时间下不了单"，等准点是正确策略 |
| 多 SKU 自动选最便宜/第一个 | 风险高（抢错版本 = 真占冤枉库存），保守 fail-fast 更稳 |
| 用 ctypes 弹 ASCII 路径警告时附带"自动迁移到 C:\BandaiSniper\"按钮 | 文件操作复杂，不如让用户手动剪切，加引导文案足够 |

## 六、风险登记

- **真扣冤枉库存**：多 SKU 误判 → 已决策 fail-fast 缓解
- **半夜误启动**：用户配错时间 → 已决策启动确认对话框缓解
- **风控触发**：concurrency / max_early_fire_ms 调激进 → 实战观察，不盲目调
- **支付超时**：抢到后用户睡过头不付 → notify 推送已有，可以加更激进提醒（如 5 分钟、10 分钟两次推送）
