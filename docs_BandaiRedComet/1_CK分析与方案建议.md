# 1. CK 分析与方案建议

> 万代南宫梦抢购脚本项目 · 前置调研
> 日期：2026-04-21
> 对应工具：`万代上号/上号软件/小程序抓Token.exe` + `Sunny.dll`

---

## 一、CK 是什么？

### 1.1 结论

**"CK" = 万代南宫梦微商城小程序登录后，服务端返回的登录凭证 Token**，放在 HTTP 请求头的 `api-access-token` 字段里。

### 1.2 证据

对 `小程序抓Token.exe` 做字符串提取，能直接看到：

| 字符串 | 含义 |
|---|---|
| `rm-app-api.bandainamcoshanghai.com/api` | 万代南宫梦小程序的后端 API 域名 |
| `api-access-token` | 登录态 Token 的请求头字段名 |
| `token----` | 工具内部给提取到的 token 打的前缀 |
| `Sunny.dll` 导入函数：`CreateSunnyNet` / `GetRequestCookie` / `SetRequestData` 等 | 说明使用 SunnyNet 做 HTTPS 中间人 |

### 1.3 命名由来

在灰产/脚本圈里 **CK = Cookie**，是"会话凭证"的统称。这个小程序其实并不用 Cookie，而是用自定义请求头 Token，但江湖叫法仍然沿用 CK。

### 1.4 风险提醒

- **拿到 CK = 拿到该微信账号在万代的登录态**，等同于账号密码的临时版本
- 一般有效期几小时到几天不等（需抓包实测）
- **绝不要登别人的号或把自己的 CK 泄露给他人**

## 二、这套工具的原理

### 2.1 架构

```
微信 PC 客户端进程
   │
   │ HTTPS 请求到 rm-app-api.bandainamcoshanghai.com
   ▼
Sunny.dll 注入 / Hook WinInet·Schannel
   │
   │ 中间人解密 TLS
   ▼
小程序抓Token.exe
   │
   │ 过滤出 api-access-token 头
   ▼
显示一串英文 → 这就是 CK
```

### 2.2 各组件说明

| 组件 | 作用 |
|---|---|
| `Sunny.dll` | [SunnyNet](https://gitee.com/qsok/sunny-net) 国产"内存级 HTTPS 中间人"代理库。**不装根证书、不走系统代理**，直接 hook 目标进程的加解密函数 |
| `小程序抓Token.exe` | e 语言写的 GUI 外壳，调用 Sunny.dll 对微信进程 MITM，过滤万代域名，抠出 `api-access-token` |
| `上号视频.mp4` | 操作演示视频 |

### 2.3 为什么要关杀毒软件

进程注入 + TLS 劫持 的手法和恶意软件完全一致，会被 AV 启发式引擎误杀。**工具本身没毒**（也可以换自己的 mitmproxy/Charles 做同样的事），但它是黑盒，不建议用来登别人的号。

## 三、对"抢购脚本"这个目标有没有帮助？

### 3.1 结论：**有，而且是整个方案的关键前置。**

### 3.2 没有 CK 的替代路径（都更差）

- **逆向 wxapkg**：解包小程序源码还原签名算法。耗时、易被反调试、一更新就失效
- **ADB + Auto.js 真机点击**：慢、不稳定、受屏幕分辨率影响、一台手机一个号

### 3.3 有 CK 之后的路径（推荐）

```
CK (api-access-token)
    ↓ 塞进 header
任何 HTTP 客户端 (Python httpx / Node axios / Go net/http)
    ↓ 直接打万代后端
rm-app-api.bandainamcoshanghai.com/api/...
```

**收益：**
- 操作时间：从人眼点击 300–500ms → 网络往返 50–100ms
- 可做并发 / 重试 / 定时触发
- 可无人值守

## 四、方案推荐（三选一）

| 方案 | 原理 | 速度 | 稳定性 | 门槛 | 适合场景 |
|---|---|---|---|---|---|
| **A. API 直调脚本** ⭐ 推荐 | 抓 CK → Python httpx 直接打接口 | 最快 (≈50ms) | 高，只要 CK 不过期 | 中，需抓一次完整下单流量 | 定点抢限量款 |
| B. Appium / Auto.js 真机点击 | 手机装 Auto.js 模拟人点 | 慢 (500ms+) | 中 | 低 | API 签名搞不定时的兜底 |
| C. 云函数 + 定时触发 | A 方案部署到 Vercel Cron / 云服务器 | 同 A | 高（公网 + 免本机在线） | 高 | 多 SKU + 无人值守 |

### 默认落地路线

> **A 方案本地跑**。如果朋友有"多账号矩阵"需求，再在 A 的基础上叠 C。

## 五、落地前必须确认的一件事

> **你/你朋友能配合抓一次"完整的人工下单流量"吗？**

因为本地环境看不到万代真实接口（只有域名和 token 头名），没有真实流量我只能写"空架子 + 明确标注待填区"的脚本。

- ✅ **能配合抓包** → 先写 PRD + 代码脚手架 + 抓包指南，抓完补完就能跑
