## 1. 背景

万代南宫梦微商城（微信小程序）不定期发售限量高达模型等热门商品，**秒空是常态**。手动点击速度不够，需要一个能：

- 在指定时间自动触发
- 直接调用万代后端 API（而非操作 UI）
- 下单成功后让用户手机端扫码支付即可

的脚本。

---

## 2. 目标与非目标

### 2.1 目标（MVP）

| # | 功能 | 说明 |
|---|---|---|
| G1 | **定时抢单** | 精确到 100ms 内触发抢购 |
| G2 | **CK 登录态** | 把抓包得到的 `api-access-token` 塞进所有请求头 |
| G3 | **单 SKU 下单闭环** | 加购 → 确认订单 → 创建订单 → 返回支付参数 |
| G4 | **失败重试** | 可配置的重试次数 + 退避策略 |
| G5 | **完整日志** | 每一步请求 / 响应 / 耗时落盘 |
| G6 | **抢购前预检** | CK 有效性 / 地址 ID / 商品可访问性 |

### 2.2 非目标（本版不做）

- ❌ 自动支付（微信支付必须手机端人工确认，脚本不做）
- ❌ 多账号矩阵（一人一机一号，后续版本）
- ❌ 商品发售日期爬取（由人手动填进配置）
- ❌ 图形界面（命令行即可）
- ❌ 绕过万代风控（不做设备模拟、代理池、机器学习对抗）
- ❌ 云端部署（V1 只求本机能跑；Vercel Cron 放进 Roadmap）

## 5. 功能需求

### 5.1 配置模块

- 使用 YAML 配置文件
- 支持 `config.example.yaml` → `config.yaml` 复制填值
- 启动时严格 schema 校验，缺项立即报错

**必填字段：**
```yaml
ck: "eyJhbGc..."            # api-access-token
timezone: "Asia/Shanghai"
snipe_time: "2026-05-01 10:00:00.000"
target:
  sku_id: "xxx"             # 从抓包得到
  quantity: 1
  address_id: "xxx"         # 预先在小程序内设置的收货地址 ID
strategy:
  pre_warmup_seconds: 60    # 抢前多少秒开始预热
  poll_interval_ms: 200     # 预热期轮询间隔
  max_retries: 10
  concurrency: 3            # 并发下单请求数
notify:
  enabled: true
  provider: "server_chan"   # 可选
  token: "SCT..."
```

### 5.2 CK 管理

- 启动时校验 CK：调用一个无副作用的接口（如 `GET /api/user/info`），200 即有效
- CK 过期立即报错并退出，**不尝试自动刷新**（小程序的 CK 刷新需要微信侧 wx.login，脚本做不到）

### 5.3 授时模块

- 支持 NTP 校时（`ntp.aliyun.com`），启动时打印本机与 NTP 偏差
- 倒计时用 `time.monotonic()`，避免系统时钟跳变
- 提前 5ms 触发第一次请求，抵消网络 TTFB

### 5.4 抢购核心

```
prepare()            # T-60s: 热连接、校 CK、查地址、缓存 sku 信息
wait_until_snipe()   # T>0: 精确倒计时
fire()               # T=0: 并发执行下单流程
    ├─ add_to_cart   (若 API 需要)
    ├─ prepare_order (确认订单接口，拿到结算快照)
    └─ create_order  (创建订单，拿到订单号 / 支付参数)
confirm()            # 打印结果 + 推送
```

- 核心下单流程之间可能存在依赖（前一步返回值喂给下一步），实现时按抓包结果串起来
- **并发策略**：对同一账号同一 SKU 起 N 个 goroutine/task，**第一个成功就取消其余**
- **重试策略**：仅在网络错误 / 5xx / 明确的"库存校验中"错误码时重试；业务错误（已购买、限购、下架）立即终止

### 5.5 日志

- `logs/YYYYMMDD_HHMMSS/` 目录
  - `run.log`：人类可读日志
  - `requests.jsonl`：每条请求的 `ts / method / url / status / latency_ms / request_body / response_body`（脱敏 CK）
- 日志保留 30 天自动清理

### 5.6 通知

- MVP 支持 Server 酱 / Bark / 飞书 webhook 任一
- 成功推送含：订单号、商品名、金额、下单耗时

## 6. 非功能需求

| 维度 | 指标 |
|---|---|
| 时延 | T=0 到第一次 HTTP 请求发出 < 20ms |
| 可靠性 | 单次抢购失败后，日志里能定位到是哪一步挂了 |
| 安全 | CK 不落盘到 git；logs 里 CK 字段自动脱敏为 `****` |
| 可移植 | macOS / Linux / Windows 均能跑（Python 3.11+） |
| 可维护 | API 层与业务层解耦，抓包结果换了只需改 `api.py` |

## 7. 技术选型

| 模块 | 选型 | 理由 |
|---|---|---|
| 语言 | Python 3.11+ | 抓包工具链熟、httpx 支持 HTTP/2 |
| HTTP | `httpx` | HTTP/2 + asyncio，秒杀场景并发友好 |
| 配置 | `pyyaml` + `pydantic` v2 | schema 校验 |
| 日志 | `loguru` | 一行配置搞定文件+控制台+结构化 |
| 异步 | `asyncio` | 标准库 |
| 时间 | `ntplib` + `time.monotonic` | 校时 + 单调时钟 |

## 9. 风险与限制

| 风险 | 说明 | 缓解 |
|---|---|---|
| **法律/合规** | 可能违反万代小程序用户协议，账号有被封风险 | 使用者自知；脚本仅用于自用，不做分发 |
| **签名反爬** | 接口可能带 `sign / nonce / timestamp`，签名算法在小程序 wxapkg 里 | 抓包阶段用 mitmproxy 观察请求序列；若有复杂签名，提级到"需 V2 逆向" |
| **CK 短时效** | Token 可能只有几小时寿命 | 抢购前 10 分钟实时抓 CK 粘进配置 |
| **人机验证** | 大促可能触发滑块/短信验证 | V1 检测到验证即中断，提示手动处理；V2 考虑接入打码平台 |
| **风控封号** | 高频请求可能触发风控 | 控制并发数 ≤ 3，拉长重试间隔 |
| **支付二次确认** | 微信支付需手机侧人工扫码 | 本来就是手机付款，不是风险，只是使用者要在抢单后立刻掏手机 |
