### 意外发现：没有加购物车接口

小程序走的是**『立即购买』直下模式**，没有 `cart/add` 这类接口。抢购路径比预期短一步：

```
商品详情(spu/v2/detail/new) → 确认订单(confirmOrder) → 提交订单(createOrder) → 微信支付
```

这对狙击是**好消息**：少一跳 = 少一个失败点 = 快几十 ms。

## 4.2 ⚠️ 重大障碍：端到端加密

### 现象
抓包后发现，TLS 层虽然被 mitmproxy 解开了，但请求/响应体**还在一层客户端加密里**。

**请求体格式**：
```json
{"encryptionBodyParams": "Sg5LEMLgnfVn3vvbw3PYoSGy+oSj5udBfXAaDFrUD9Ng..."}
```

**URL 参数格式**：
```
GET /api/commodity/v1/app/spu/v2/detail/new?encryptionUrlParams=nW1FXm168kQOhYQQOm6XLw%253D%253D
```

**响应体**：整个 body 就是一大串 base64 密文（没有 JSON 外壳），例如：
```
APVSZZC5KMsWvhOIVB48vPLXgJ7/3SSmTazrP+lGiQBKyZ2AfFdWDkm6gGS6ldMV...
```

### 影响

1. **光有 CK（`api-access-token`）不够**，还必须拿到**客户端加密/解密算法**才能：
   - 按商品 ID 构造加密的 URL 参数
   - 构造 `confirmOrder` / `createOrder` 的请求体
   - 解出 `createOrder` 响应里的 `orderId` / `prepay_id`
2. **明显是业务方刻意反爬的措施**，不是常规 TLS 加密能覆盖的
3. **替身方案（Sunny.dll / SunnyNet）能绕过这层**：因为它在 WeChatAppEx 进程里 hook 的是 `wx.request` 的 **JS 层**，拿到的是加密**前**的明文 —— 这解释了 `万代上号` 工具为什么只需要 CK 就能干事

### 认证头仍然标准
`api-access-token` 是标准的 JWT 类长串（175 字符），作为 HTTP header 传入：
```http
api-access-token: REDACTED_API_ACCESS_TOKEN_len175
Content-Type: application/json
User-Agent: Mozilla/5.0 ... MicroMessenger/7.0.20.1781 ... WindowsWechat/WMPF ...
Referer: https://servicewechat.com/wx1cb4557915b2b7cd/129/page-frame.html
```

注：`wx1cb4557915b2b7cd` 是小程序 AppID，`129` 是当前版本号。

## 4.3 下一步：三条路对比

| 方案 | 做法 | 工作量 | 最终效果 |
|---|---|---|---|
| **A. 逆向 .wxapkg** | 从微信本地缓存（`%USERPROFILE%\Documents\WeChat Files\Applet\wx1cb4557915b2b7cd\` 或 `AppData\Roaming\Tencent\xwechat\radium\...`）找到 `__APP__.wxapkg`，用 `wxappUnpacker` 解开得到 JS 源码，grep `encryptionBodyParams` 定位到加密函数 —— 通常是 AES-CBC/ECB + 硬编码密钥 | 中（2–4 小时） | ✅ 一劳永逸，Python 端直接构造合法请求，走 API 层最快 |
| **B. UI 自动化** | 用 pywinauto 或 `uiautomation` 库，在微信 PC 窗口内模拟点击『立即购买 → 提交订单』；或上安卓模拟器 + ADB | 低（0.5–1 天） | ⚠️ 比人快但比 API 慢（300–800 ms 级 vs 50–100 ms 级），热门品类里不一定能抢到 |
| **C. 拟 SunnyNet hook** | 写 .NET 注入 DLL，把 `wx.request` 的参数在序列化前截获 | 高（2–5 天） | ✅ 最本质，但开发成本高 |

### 本项目选择：A

**理由**：
- 抢购脚本的核心价值是**速度**。A 是唯一能把响应时间压到 ~100 ms 以内的方案
- A 的工作量是一次性的。密钥如果是硬编码（大概率），后续所有接口都通
- 失败兜底：A 如果走不通（密钥是服务端下发 / 动态加密），再回退到 B

## 4.4 A 路线工作量预研（2026-04-21 晚）

### 4.4.1 缓存路径（微信 4.0+）

用户机器上老路径 `Documents\WeChat Files\Applet\` 不存在。经查，微信 4.0+ 改用 xwechat/radium 架构，`.wxapkg` 缓存在：

```
%APPDATA%\Tencent\xwechat\radium\Applet\packages\<wxid>\<版本号>\__APP__.wxapkg
```

完整路径示例：
```
C:\Users\<用户>\AppData\Roaming\Tencent\xwechat\radium\Applet\packages\wx1cb4557915b2b7cd\129\__APP__.wxapkg
```

可能还有 `__SUBPACKAGE__*.wxapkg`（子包）。

**查找脚本**：`tools/unpack/find_wxapkg.ps1`

### 4.4.2 PC 端 wxapkg 额外加密（V1MMWX）

微信 PC 版的 wxapkg 文件**比手机端多一层加密**（文件头 `V1MMWX`）。解密算法：

1. `key = PBKDF2(password=wxid, salt="saltiest", iter=1000, dkLen=32)`
2. 去掉 6 字节 `V1MMWX` 头后，前 1023 字节 AES-128-CBC 解密，IV = `"the iv: 16 bytes"`
3. 剩余字节 XOR `wxid[-2]`（AppID 倒数第二个字符）
4. 结果以 `0xBE` 开头 = 标准 wxapkg，喂给常规解包器

**好消息**：成熟工具已经封装好了这一步，**不用手搓**。

### 4.4.3 推荐工具：`wux1an/wxapkg`（Go）

2025/2026 年最活跃的开源解包器（3k+ stars），一条命令搞定"扫描 → V1MMWX 解密 → 解包 → JS/WXML/JSON 美化"。

- 仓库：https://github.com/wux1an/wxapkg
- 安装：下载 Releases 里的 prebuilt 二进制，或 `go install github.com/wux1an/wxapkg@latest`
- 用法：`wxapkg scan`（自动扫目标）或 `wxapkg unpack -o out/ __APP__.wxapkg`

备选：`halo951/mp-wxapkg-unpacker`（Node）、`zhuweiyou/wxapkg`（Node 拖拽 .exe）。

### 4.4.4 加密函数定位策略

解包后得到 `app-service.js`、`app.js`、`workers.js` 以及各页面的 JS。在这些里 grep：

**字符串关键词**：
- `encryptionBodyParams` / `encryptionUrlParams`
- `encryptBody` / `decryptBody` / `aesEncrypt` / `aesDecrypt`
- `CryptoJS` / `WordArray` / `Pkcs7` / `enc.Base64` / `enc.Utf8`

**库特征**：多半是 `crypto-js` 的 `CryptoJS.AES.encrypt(..., key, { iv, mode, padding })`。少数自带 AES（看 SBOX 常量 `0x637c777b…`）。

**密钥位置**：通常是 16/24/32 字符的 ASCII 字符串字面量，硬编码在加密函数附近。正则：
```regex
['"][A-Za-z0-9+/=]{16}['"]
```

**入口点**：搜 `wx.request` 的拦截器包装，一般叫 `requestInterceptor` / `httpUtil` / `http.js`，那里会重写 body 成 `{encryptionBodyParams: ...}`。

### 4.4.5 时间预估

假设 **JS 代码没有恶意混淆**（小程序通常没有）：

| 情况 | 耗时 |
|---|---|
| 🟢 乐观（密钥是明文字符串，就在 `CryptoJS.AES.encrypt` 调用旁边） | ~1.5 h |
| 🟡 典型（需要少量美化/重命名，追一下拦截器链，可能有 IV 推导小花样） | 4–6 h |
| 🔴 悲观（自定义 AES / 密钥从运行时状态派生 / 签名+加密双层 / 找错子包） | 12–20 h |

路径发现 + 首次解包成功：另加 ~30 min，工具成熟，基本都能一把过。

**结论**：**预计 4–6 小时可搞定** A 路线。值得做。
