### 意外发现：没有加购物车接口

小程序走的是**『立即购买』直下模式**，没有 `cart/add` 这类接口。抢购路径比预期短一步：

```
商品详情(spu/v2/detail/new) → 确认订单(confirmOrder) → 提交订单(createOrder) → 微信支付
```

这对狙击是**好消息**：少一跳 = 少一个失败点 = 快几十 ms。

## 4.2 ⚠️ 重大障碍：端到端加密

### 现象
抓包后发现，TLS 层虽然被 mitmproxy 解开了，但请求/响应体**还在一层客户端加密里**。

**请求体格式**：
```json
{"encryptionBodyParams": "Sg5LEMLgnfVn3vvbw3PYoSGy+oSj5udBfXAaDFrUD9Ng..."}
```

**URL 参数格式**：
```
GET /api/commodity/v1/app/spu/v2/detail/new?encryptionUrlParams=nW1FXm168kQOhYQQOm6XLw%253D%253D
```

**响应体**：整个 body 就是一大串 base64 密文（没有 JSON 外壳），例如：
```
APVSZZC5KMsWvhOIVB48vPLXgJ7/3SSmTazrP+lGiQBKyZ2AfFdWDkm6gGS6ldMV...
```

### 影响

1. **光有 CK（`api-access-token`）不够**，还必须拿到**客户端加密/解密算法**才能：
   - 按商品 ID 构造加密的 URL 参数
   - 构造 `confirmOrder` / `createOrder` 的请求体
   - 解出 `createOrder` 响应里的 `orderId` / `prepay_id`
2. **明显是业务方刻意反爬的措施**，不是常规 TLS 加密能覆盖的
3. **替身方案（Sunny.dll / SunnyNet）能绕过这层**：因为它在 WeChatAppEx 进程里 hook 的是 `wx.request` 的 **JS 层**，拿到的是加密**前**的明文 —— 这解释了 `万代上号` 工具为什么只需要 CK 就能干事

### 认证头仍然标准
`api-access-token` 是标准的 JWT 类长串（175 字符），作为 HTTP header 传入：
```http
api-access-token: REDACTED_API_ACCESS_TOKEN_len175
Content-Type: application/json
User-Agent: Mozilla/5.0 ... MicroMessenger/7.0.20.1781 ... WindowsWechat/WMPF ...
Referer: https://servicewechat.com/wx1cb4557915b2b7cd/129/page-frame.html
```

注：`wx1cb4557915b2b7cd` 是小程序 AppID，`129` 是当前版本号。

## 4.3 下一步：三条路对比

| 方案 | 做法 | 工作量 | 最终效果 |
|---|---|---|---|
| **A. 逆向 .wxapkg** | 从微信本地缓存（`%USERPROFILE%\Documents\WeChat Files\Applet\wx1cb4557915b2b7cd\` 或 `AppData\Roaming\Tencent\xwechat\radium\...`）找到 `__APP__.wxapkg`，用 `wxappUnpacker` 解开得到 JS 源码，grep `encryptionBodyParams` 定位到加密函数 —— 通常是 AES-CBC/ECB + 硬编码密钥 | 中（2–4 小时） | ✅ 一劳永逸，Python 端直接构造合法请求，走 API 层最快 |
| **B. UI 自动化** | 用 pywinauto 或 `uiautomation` 库，在微信 PC 窗口内模拟点击『立即购买 → 提交订单』；或上安卓模拟器 + ADB | 低（0.5–1 天） | ⚠️ 比人快但比 API 慢（300–800 ms 级 vs 50–100 ms 级），热门品类里不一定能抢到 |
| **C. 拟 SunnyNet hook** | 写 .NET 注入 DLL，把 `wx.request` 的参数在序列化前截获 | 高（2–5 天） | ✅ 最本质，但开发成本高 |

### 本项目选择：A

**理由**：
- 抢购脚本的核心价值是**速度**。A 是唯一能把响应时间压到 ~100 ms 以内的方案
- A 的工作量是一次性的。密钥如果是硬编码（大概率），后续所有接口都通
- 失败兜底：A 如果走不通（密钥是服务端下发 / 动态加密），再回退到 B

```
或
```
%USERPROFILE%\Documents\xwechat_files\radium\...
