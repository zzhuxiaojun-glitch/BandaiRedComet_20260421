# 5. 方案 B：UI 自动化 规划

**创建日期**：2026-04-21
**定位**：**Plan A（API 直连）的兜底方案**，不是替代

---

## 5.0 为什么还需要 Plan B

虽然 Plan A 已经推翻了"绕不过去接口"的前提（密钥、算法、请求结构全拿到），但 Plan A 有**三个潜在失效点**：

| 风险 | 具体场景 | 概率 |
|---|---|---|
| 🔴 服务端风控 | 检测到签名合法但 UA / 网络指纹不像真微信客户端，直接拒绝或限流 | 中 |
| 🟡 版本迭代失效 | 万代下次更新小程序换了密钥或多加一层签名，Plan A 脚本当天废掉 | 中 |
| 🟡 账号安全 | API 直刷被识别为"机器"，账号被风控标记（限购/禁购） | 中 |

**Plan B 的价值**：
- **完全绕过 API 层风控**（就是真用户真点击，服务端只看到合法 WebSocket 帧）
- 失效概率更低（UI 变动是看得见的，API 变动是看不见的）
- 账号更干净（操作轨迹和人类无异）

**代价**：慢 —— 典型 300–800 ms/次点击 vs Plan A 的 50–100 ms/次请求。热门瞬爆款 Plan A 赢，冷一点的 Plan B 足够。

## 5.1 朋友给的 4 条原则（完全正确，A/B 都适用）

| 原则 | 我们怎么落实 |
|---|---|
| **1. 提前准备好一切**：商品页提前打开、SKU/地址/支付选好，开抢只剩"点一下" | 脚本启动后自动打开商品页→选中目标 SKU→地址/支付方式复用默认配置。到开抢时间前 5 秒，页面就停在"去结算"一步前 |
| **2. 时间要准**：对齐服务器时间，差几十毫秒就输了 | NTP 同步 + 调用 `/api/base/getTimestamp` 校准本地钟（见 `src/bandai_sniper/timer.py`） |
| **3. 路径要短**：一点直接到下单/支付，不要中间弹窗选规格 | 提前把『选规格/选地址/选支付』全做完，临点开抢只剩最后一下『提交订单 → 确认支付』 |
| **4. 稳定比花活**：好网络、别频繁切号/切IP | 有线/5G 独占网络；同一账号稳定使用，不并发多账号；UA/IP 指纹保持一致 |

## 5.2 Plan B 技术选型

### 方案 B1：Windows 微信 PC + pywinauto / uiautomation

**原理**：通过 Windows 辅助功能 API 识别微信窗口里的按钮、输入框，模拟点击/输入。

- **库**：`uiautomation`（国人写的，对中文控件识别好）> `pywinauto`
- **优点**：不用模拟器，就在用户当前微信里操作
- **缺点**：微信小程序的 UI 控件对辅助功能 API **大多不暴露文本**，只能靠坐标/截图找色
- **实际可行性**：⚠️ 中等 —— 小程序不是原生 Windows 控件，是 Chromium WebView，`uiautomation` 看到的只是一堆 `Pane` 或空控件。纯坐标点击 + 截图识别按钮位置是更现实的做法

### 方案 B2：安卓模拟器 + ADB

**原理**：在 MuMu / 雷电 / Genymotion 模拟器里装微信手机版，通过 ADB `input tap` / `uiautomator dump` 操作。

- **库**：`uiautomator2`（Python 封装，activity 感知）
- **优点**：ADB 指令稳、UI 层级可解析、录屏录像方便
- **缺点**：模拟器启动慢、微信手机版对模拟器有风控
- **实际可行性**：✅ 高 —— 开源社区此类自动化最多，资料全

### 方案 B3：OpenCV 图像识别 + pyautogui

**原理**：截取当前屏幕，用 OpenCV 模板匹配找『提交订单』按钮位置，`pyautogui.click(x, y)` 点它。

- **优点**：**跨平台，不依赖控件树**，只靠视觉
- **缺点**：按钮图样变了就失效；分辨率/缩放敏感
- **实际可行性**：✅ 高 —— 逻辑简单，适合做保底

### 推荐组合

**B1 + B3**：Windows 微信 + uiautomation 定位小程序窗口边界，窗口内用 OpenCV 模板匹配按钮位置。
- 保留用户自己的账号与网络环境（第 4 条）
- 不用启模拟器（启动慢）
- 图像识别稳 —— 万代小程序 UI 风格统一，按钮图案不会常变

## 5.3 Plan B 架构（B1 + B3 路线）

```
┌─────────────────────────────────────────────────────────────┐
│                  sniper.py （总调度）                        │
│                                                              │
│  Plan A 主路径  ──会失败──→  Plan B fallback                  │
│   ↓                           ↓                              │
│  API 直连下单                 触发 UI 自动化                  │
└─────────────────────────────────────────────────────────────┘
                                  │
                                  ▼
┌──────────────────────────────────────────────────────────────┐
│           bandai_sniper/ui_bot.py (新增)                      │
│                                                              │
│  1. 找到微信窗口句柄（uiautomation.WindowControl）             │
│  2. 激活窗口 → 跳转到已预热的小程序商品页                       │
│  3. 循环截屏 → OpenCV 模板匹配『立即购买』按钮                  │
│  4. 匹配到 → pyautogui.click()                                │
│  5. 等下一屏 → 匹配『提交订单』                                │
│  6. 等下一屏 → 匹配『确认支付』                                │
│  7. 微信支付密码：由用户手动输入（不自动化，避免安全风险）       │
│                                                              │
│  核心循环间隔：20 ms 截屏 + 10 ms 匹配 = 30 ms 周期             │
└──────────────────────────────────────────────────────────────┘
```

**关键取舍**：
- **支付密码不自动化** —— 支付环节保留人工输入，既避免账号盗用风险，也符合微信支付的合规要求
- **截图模板提前录好** —— 项目自带 `assets/ui_templates/` 目录，内含『立即购买/提交订单/确认支付』三张基准图
- **坐标缩放自适应** —— 用 `cv2.matchTemplate` + `TM_CCOEFF_NORMED` 阈值 0.85，兼容 100%/125%/150% DPI

## 5.4 Plan B 任务分解（工作量估算）

| 任务 | 耗时 | 备注 |
|---|---|---|
| 调研 + 搭脚手架（uiautomation / pyautogui / opencv-python 装好） | 1 h | 已经有 Python 3.11 + venv |
| 录制按钮模板图（『立即购买/提交订单/确认支付』三张） | 30 min | 在本机不同 DPI 下各录一套 |
| 窗口定位 + 小程序页面聚焦 | 1 h | uiautomation 爬微信窗口树 |
| 循环匹配 + 点击逻辑 | 2 h | 含失败重试、超时保护、日志 |
| NTP 对时 + 定时触发 | 复用 Plan A 的 `timer.py` | 0 h |
| 端到端联调（1 笔真实下单） | 1 h | 用便宜非抢购商品做冒烟 |
| **合计** | **5–6 h** | |

**总产出**：`src/bandai_sniper/ui_bot.py`（约 300 行）+ `assets/ui_templates/*.png`

## 5.5 A + B 融合策略

实战时两条路**同时准备，按商品热度选武器**：

| 商品类型 | 主用 | 备用 |
|---|---|---|
| 🔥 瞬爆款（0 秒抢光） | Plan A（必须 API 最快） | Plan B 作异常兜底 |
| 🟡 中热度（几十秒内抢光） | Plan A 或 B 都行 | — |
| 🟢 低热度（几分钟） | Plan B（更稳） | — |

**自动降级**（未来可做）：
- Plan A 调用 `createOrder` 被风控拒绝（返回码非 0、连续超时）
- 主程序自动切到 Plan B，将用户当前微信窗口激活、截屏找按钮、继续完成下单
- 用户感知：后台悄无声息换了武器

## 5.6 现阶段决定

**当前优先**：**先把 Plan A 端到端跑通**（签名、时间戳是否被服务端认），如果通了就进入实战压测。Plan A 如果在实战阶段被风控拒绝，**再按本文档启动 Plan B**。

**不立即并行**的理由：
- 两路并行会让主程序变复杂，目前不需要
- Plan A 刚突破，未经实战验证。先专注把它跑通
- Plan B 本身是独立的 5–6 h 工作量，需要时再投

**如果朋友坚持想要 Plan B**：
- 可以把 Plan B 作为**小程序风控触发后的保底路线**（A 挂了再启 B）
- 也可以**只做 Plan B、跳过 Plan A**（如果你觉得账号安全/封号风险比速度更重要）
