# 9. 踩坑汇总（2026-04-21 ~ 22）

**用途**：项目中遇到过的所有非显而易见的陷阱。下次遇到类似场景直接查，不要重踩。

---

## 9.1 加密/签名链路（最硬核）

### 坑 1：AES key 长得像 base64 但**不是**

- **现象**：`secretKey = "5p/9OgW9F5cJc0fOs0TXYA=="` 末尾 `==` 符号，第一反应想 `b64decode` 一下再当 16 字节 AES key 用
- **真相**：JS 源码里是 `CryptoJS.enc.Utf8.parse(secretKey)` —— **把这个字符串当 UTF-8 字面量**，就是 24 个 ASCII 字节当 key 用，AES-192-ECB（不是 128）
- **教训**：看见像 base64 的 key，先去源码里看它怎么被 `.parse()`，别想当然

---

### 坑 2：JS `o(e, r, t)` 的参数含义误读 🔴

- **现象**：自己读 JS 源码时把 `t` 当成 sign/encrypt 开关，结果所有嵌套 Object/Array 都被错误地包成 `{"str": JSON.stringify(...)}`，12 个样本只对 1 个
- **真相**（读完全源码才明白）：
  - `t` = **`encryptionEnable`**（线上恒 true）
  - `r` = sign(false) / encrypt(true) 开关
  - 对顶层 Array：sign 路径 `u = {str: JSON.stringify(原数组)}`，encrypt 路径深拷贝原数组
  - 对嵌套 Object/Array：**sign 和 encrypt 两条路径完全一致**，都让 replacer 递归把 primitive 叶子转字符串
- **教训**：多参数函数的语义不要靠单样本推，一定要读完整代码

---

### 坑 3：sign 路径的 inner JSON **不走 replacer** 🔴

- **现象**：顶层入参是纯数字数组 `[6, 10]`（subscribe 的 getTemplateId），sign 签名对不上
- **真相**：JS 里 `u = {str: JSON.stringify(e)}` 这里 `JSON.stringify(e)` **不传 replacer** —— 所以数字保留原样 `"[6,10]"`，不转字符串
- **错误做法**：先 `_coerce_leaves` 把数字转字符串再 `json.dumps` → 产出 `'{"str":"[\"6\",\"10\"]"}'`
- **正确做法**：sign 模式下对**原数组**做 `json.dumps`（separators=`(",", ":")`）
- **踩时机**：数字数组入参才会暴露，纯字符串数组两种方式刚好一致
- **Python float 额外注意**：Python `json.dumps(142.0)` = `"142.0"`，JS `JSON.stringify(142.0)` = `"142"`。需要写 `_js_json_dumps_raw` 先把整数值 float 转 int

---

### 坑 4：createOrder 入参所有数字**必须用 JS 风格字符串化**

- **现象**：createOrder 入参是 confirmOrder 响应 data 原样回传，但 Python `str(142.0)` = `"142.0"`，HAR 里真实是 `"142"`
- **原因**：JS 没有 int/float 之分，`String(142.0) === "142"`；Python 会保留 `.0`
- **修复**：写 `_js_str(v)` 辅助：
  ```python
  def _js_str(v):
      if isinstance(v, float) and v.is_integer():
          return str(int(v))
      return str(v)
  ```
- **教训**：跨语言复现逻辑，小心 number ↔ string 的字面量差异

---

### 坑 5：JS md5 实现的 **UTF-16 截断 bug** ⭐ 最坑

- **现象**：13 个接口签名离线验证 12/12 通过，但下单抓 HAR 后，所有**参数含中文**的请求（confirmOrder 带地址、createOrder）签名对不上
- **真相**：源码 `F86FDFF6...js` 是一个老式手搓 md5（约 2001 年风格）：
  ```js
  t[n >> 5] |= (255 & r.charCodeAt(n / 8)) << n % 32;
  ```
  对 JS 字符串的每个字符取 `charCodeAt & 0xFF` —— **UTF-16 码点强制截成低 8 位字节**。ASCII 无影响，中文字符被废掉：
  - `上` (U+4E0A) → `0x0A`（换行）
  - `海` (U+6D77) → `0x77`（'w'）
- **Python 复现**：
  ```python
  hashlib.md5(bytes(ord(c) & 0xFF for c in combo)).hexdigest()
  ```
  **不是** `combo.encode("utf-8")`
- **为啥第一次没暴露**：第一次抓的 HAR 全是浏览流，参数都是英文数字，纯 ASCII 无差别。第二次抓下单流带地址（`"cityName":"上海市"` 等）才触发
- **教训**：验证签名算法必须覆盖**含中文参数**的 case，否则会误以为 ASCII-only 的正确算法就是对的

---

### 坑 6：HAR 响应体整块就是 AES 密文（无 JSON 外壳）

- **现象**：初版 dump 脚本把响应当 `{"code":0,"data":"<密文>"}` 结构解析，全部失败
- **真相**：响应 body 整块就是 base64 AES 密文，`aes_decrypt(resp.text.strip())` 才能得到合法 JSON
- **教训**：客户端加密的 API 有可能响应也是纯密文（不一定是 "code/msg 外壳 + data 密文" 模式）

---

### 坑 7：`encryptionUrlParams` 在 HAR 里可能双层 URL 编码

- **现象**：GET 请求的 URL 参数里出现 `%253D%253D`，直接 base64 decode 报 `Incorrect padding`
- **真相**：抓包工具或浏览器对 URL 又叠了一层编码
- **修复**：解密前 `unquote` 一次，不行再 `unquote(unquote(...))` —— dump 脚本里做三重尝试
- **教训**：URL 参数的密文在 HAR 里不一定是"发送时的那个字符串"

---

## 9.2 下单流程字段的坑

### 坑 8：地址接口路径在 `/order/v1/app/address/` 不在 `/member/`

- **现象**：直觉会以为地址归用户模块，写 `/api/member/.../address/list` 一查 404
- **真相**：万代把地址接口归到订单模块：
  - `GET /api/order/v1/app/address/getDefaultOrderAddress`
  - `POST /api/order/v1/app/address/getOrderAddressList`

---

### 坑 9：`spu/sku/detail` 参数名 `id`，但填的是 **spuId**

- **现象**：接口 `GET /api/commodity/v1/app/spu/sku/detail?id=6521`，参数叫 `id` 容易被当 skuId
- **真相**：这个 `id` 是 **spuId**，返回的是该 SPU 下的 SKU 列表
- **教训**：参数名别想当然，对照 HAR 真实值看

---

### 坑 10：createOrder 响应字段 `packagev` 不是 `package`

- **现象**：返回里有 `"packagev": "prepay_id=wx22..."`
- **真相**：微信支付 JSAPI 标准字段名是 `package`（5 个字母），这里后端写成 `packagev`（疑似 typo 或故意混淆）
- **前端怎么用**：小程序里手动把 `packagev` 映射回 `package` 再传给 `wx.requestPayment`
- **代码做法**：`PayParams.from_resp` 里 `package=data.get("package") or data["packagev"]` 双保险

---

### 坑 11：createOrder 入参不是"简短版" —— 是 confirmOrder 响应的**字符串化完整回传**

- **现象**：想当然以为 `createOrder({skuList, addressId})` 就够
- **真相**：入参要把 confirmOrder 响应里的 `orderAmount / goodsPrice / freight / balanceAmount / depositAmount / orderPoint / ...` 全部原样回传（数字全转字符串），外加完整地址字段（不是 addressId，是 province/city/district/receiver 等字段展开）+ `usePoint`
- **教训**：涉及复杂计算的下单接口，**先把 confirm 响应收好直接回传**，别自己造字段

---

### 坑 12：`latitude`/`longitude` 真的会传字符串 `"undefined"`

- **现象**：HAR 里看到 `"latitude": "undefined"` 字面量
- **真相**：小程序代码里 `m.latitude = p.latitude`，`p.latitude` 是 `undefined`（JS 的 undefined），在后续 `l(i)` 处理不应该保留，但在某些代码路径里被 `String(undefined)` 转成了字符串 `"undefined"` 先进了 params，然后 `l(i)` 对非空字符串不删
- **做法**：直接照抄，入参写死 `"latitude": "undefined", "longitude": "undefined"`
- **不踩的话**：自己构造传 `None` 或干脆不传，**签名会对不上**（因为真实 params 里有这两个键）

---

## 9.3 工具链 / 抓包的坑

### 坑 13：`bandai_filter.py` 误杀微信其他域的流量

- **现象**：mitmproxy 启动后，微信打开其他小程序或聊天都网络异常
- **原因**：初版写了自定义过滤脚本，想让非万代域"穿透"，但 mitmproxy 的中间人已经劫持了 TCP，自定义脚本没正确放行
- **修复**：换成 mitmproxy 内置 `--set allow_hosts=.*\.bandainamcoshanghai\.com` —— 非目标域根本不 MITM，原样透传
- **教训**：别和 mitmproxy 的 filter 机制斗，用内置 `allow_hosts` / `ignore_hosts` 最省事

---

### 坑 14：Python venv 偶发静默失败

- **现象**：`python3 -m venv .venv` 返回 0 但 `.venv/` 没完整创建
- **修复**：删掉残留目录，重跑一次
- **教训**：venv 创建完马上 `.venv/bin/python --version` 确认，别直接 `pip install`

---

### 坑 15：PowerShell 路径含 `&` 字符报 `AmpersandNotAllowed`

- **现象**：`cd C:\…\【自开发_代码&发布版win仓库】` 失败
- **修复**：双引号包裹整个路径 `cd "C:\...\【自开发_代码&发布版win仓库】"`
- **教训**：PowerShell 对 `&` 很敏感（它是调用操作符），凡路径里带 shell 特殊字符必须引号

---

## 9.4 环境 / 平台的坑

### 坑 16：WSL 读 Windows 盘路径的编码问题

- **现象**：`ls /mnt/c/2025.10.16_move/IVIS_new/IT相关/【md_repo】/` 有些时候显示乱码
- **状况**：中文 + `【】` 全角括号在 bash 里可直接用，但在 heredoc 或环境变量里需要小心引号层级
- **做法**：路径参数传递时用**双引号整段包**，避免转义

---

### 坑 17：微信小程序 4.0+ 的 wxapkg 存储位置变了

- **现象**：搜老版本文档里说的 `WeChat Files\Applet\` 找不到
- **真相**：微信 4.x 改用 radium 架构：
  ```
  %APPDATA%\Tencent\xwechat\radium\users\<uuid>\applet\packages\<wxid>\<version>\__APP__.wxapkg
  ```
- **教训**：wxapkg 查找要按当前微信版本的架构找，老教程可能失效

---

## 9.5 总结 / 回顾

**最贵的坑排行**（按耗时）：
1. JS md5 UTF-16 截断 bug（坑 5）—— 花了 ~1.5h 排查
2. JS `o()` 参数误读（坑 2）—— 花了 ~1h 读源码才明白
3. bandai_filter.py 误杀流量（坑 13）—— 花了 ~0.5h 换方案

**预防经验**：
- 跨语言复现算法，务必在**各种边界 case** 都验证（ASCII vs 非 ASCII、int vs float、空 vs 非空、嵌套 vs 扁平）
- 对"接口响应格式"别靠文档猜，直接解密真实 HAR 看完整字段
- 对"字段名"别靠直觉（packagev vs package / id as spuId），对照真实请求
- 签名算法挂了别瞎改盐或公式，优先**逐字节对比** Python 生成的输入和服务端应该收到的输入
